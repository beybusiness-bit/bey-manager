Chapter 1 · 기초편
웹 앱은 어떻게 생겼나요?
웹 앱을 만들기 전에 전체 구조를 이해해두면 클로드에게 요청하고 설명하기가 훨씬 쉬워져요.
프론트엔드 (화면)꼭 알기

사용자 눈에 보이는 모든 것이에요. 버튼, 텍스트, 이미지, 입력창. HTML/CSS/JavaScript로 만들어지고 사용자의 브라우저에서 실행돼요.

식당의 홀 — 손님이 직접 보고 앉고 메뉴를 고르는 공간이에요.
백엔드 (서버)알면 좋음

사용자 눈에 안 보이지만 실제 계산·저장·로그인 처리가 일어나는 곳이에요. Python, Node.js 같은 언어로 만들고 서버 컴퓨터에서 실행돼요.

식당의 주방 — 손님은 못 들어가지만 음식이 만들어지는 곳이에요.
데이터베이스 (DB)알면 좋음

데이터를 저장하는 창고예요. 회원 정보, 게시글, 주문 내역 등이 여기에 저장돼요. 간단한 앱은 Google Sheets를 DB 대용으로 쓰기도 해요.

식당의 냉장고 — 재료(데이터)를 보관해두고 필요할 때 꺼내요.
클라이언트 vs 서버꼭 알기

클라이언트는 요청하는 쪽(브라우저, 앱)이고, 서버는 응답하는 쪽이에요. "클라이언트 사이드 처리" = 서버 없이 브라우저에서만 실행된다는 뜻이에요.

클로드가 "클라이언트에서 처리해요" 하면 = 내 브라우저에서만 동작하는 코드라는 뜻이에요. 서버 비용이 안 들어요.
Chapter 2 · 기초편
개발에서 자주 쓰는 언어들
언어 이름만 들어도 막막하죠. 각각 어떤 역할인지만 알면 충분해요.
HTML — 뼈대꼭 알기

"여기 제목", "여기 버튼", "여기 이미지" 구조를 잡아요. 꺾쇠(< >)로 감싸는 태그를 씁니다.

<!-- 주석: 이런 식으로 생겼어요 --> <h1>안녕하세요</h1> <button>클릭하기</button>
CSS — 디자인꼭 알기

색, 폰트, 크기, 여백, 버튼 모양을 정해요. HTML이 뼈대라면 CSS는 옷이에요.

button { background-color: blue; border-radius: 8px; }
JavaScript (JS) — 동작꼭 알기

버튼 클릭 반응, 데이터 계산, 서버 통신이 JS로 이루어져요. 세 언어 중 가장 복잡하지만 클로드가 전부 써줘요.

"JS", "자바스크립트", "스크립트"는 모두 같은 말이에요. Java와는 완전히 다른 언어예요 — 이름만 비슷해요!
TypeScript (TS)나중에

JavaScript에 타입 체크 기능을 추가한 언어예요. 클로드가 TS로 코드를 쓰기도 해요. "JS랑 거의 같은데 더 엄격한 것"으로 이해하면 충분해요.

클로드가 .ts 파일을 만들었다면 TypeScript예요. 당황하지 말아요 — JS처럼 쓰면 돼요.
Chapter 3 · 실전편
클로드가 준 코드, 어떻게 읽나요?
직접 짤 줄 몰라도 구조를 눈으로 읽을 수 있으면 대화가 훨씬 쉬워져요.
함수 (function)핵심 개념

어떤 작업을 묶어서 이름을 붙인 것이에요. ()가 붙어 있으면 함수 호출이에요.

function greet(name) { return "안녕, " + name; } greet("베이"); // → "안녕, 베이"
레시피 카드. 이름을 붙여두고 필요할 때마다 꺼내 써요.
변수 (variable)핵심 개념

값을 담아두는 이름표예요. const는 변경 불가, let은 변경 가능.

const appName = "베이 앱"; // 안 바뀜 let count = 0; // 바뀔 수 있음 count = count + 1; // 이제 1
조건문 (if)핵심 개념

"만약 ~이면 이렇게"를 표현해요. 거의 모든 코드에 등장해요.

if (isLoggedIn) { showDashboard(); } else { showLoginPage(); }
화살표 함수 (Arrow Function)자주 보임

함수를 짧게 쓰는 문법이에요. 클로드가 자주 씁니다.

// 일반 함수와 같은 동작이에요 const add = (a, b) => a + b; // 또는 items.map(item => item.name)
=> 가 보이면 "아, 함수구나"라고 인식하면 돼요.
Chapter 4 · 실전편
에러 메시지 해석하기
빨간 글씨가 떠도 당황하지 말아요. 패턴만 알면 겁이 훨씬 덜 나요.
에러 메시지 구조핵심 패턴
에러 종류: 뭐가 잘못됐는지 (예: TypeError)
에러 내용: 구체적으로 어떤 문제인지
위치: 몇 번째 줄에서 났는지 (at line 23)
전부 이해 안 해도 돼요. 에러 메시지 전체를 복사해서 클로드에게 주면 돼요.
자주 보는 에러 종류참고용

is not defined → 없는 변수·함수를 썼어요
Cannot read properties of undefined → 값이 없는데 뭔가 꺼내려 했어요
SyntaxError → 괄호가 안 닫혔거나 오타
404 Not Found → 요청한 주소·파일이 없어요
401 Unauthorized → 로그인 필요 or 권한 없음
500 Internal Server Error → 서버 쪽 문제

콘솔(Console) 여는 법꼭 알기

Windows/Linux: F12 → Console 탭
Mac: Cmd + Option + J

앱이 이상하면 콘솔부터 열어요. 빨간 글씨 = 에러, 노란 글씨 = 경고예요.
"빌드 실패" vs "런타임 에러"구분 팁

빌드 에러: 앱을 시작하기 전에 나는 에러. 문법 오류가 많아요. 터미널에 떠요.
런타임 에러: 앱이 실행 중에 나는 에러. 특정 동작을 할 때 터져요. 콘솔에 떠요.

터미널이 빨개서 앱이 아예 안 켜진다 = 빌드 에러. 앱은 켜졌는데 버튼 누를 때 터진다 = 런타임 에러.
Chapter 5 · 실전편
파일, 폴더 구조와 배포
프로젝트 파일 구조를 알면 클로드가 뭘 만들고 있는지 훨씬 쉽게 파악돼요.
기본 웹앱 파일 구조꼭 알기
my-app/ ├── index.html # 시작 페이지 ├── style.css # 디자인 ├── script.js # 기능 └── images/ # 이미지 폴더
단일 HTML 파일 앱은 CSS·JS를 HTML 안에 전부 넣기도 해요. 간단한 앱에 많이 쓰이는 방식이에요.
로컬 vs 배포꼭 알기

로컬(local): 내 컴퓨터에서만 돌아가요. 주소가 localhost:3000처럼 생겼어요.

배포(deploy): 인터넷 서버에 올려서 누구나 접속 가능. 실제 URL이 생겨요.

"로컬에서는 되는데 배포 후에 안 돼요"는 매우 흔한 상황이에요. 환경이 달라서 생기는 문제예요.
무료 배포 도구꼭 알기

GitHub Pages: HTML 파일 무료 배포. 정적 사이트에 딱이에요.
Vercel / Netlify: React 같은 복잡한 앱도 무료 배포 가능. 더 추천.
Replit: 코딩 + 배포를 한 곳에서.

Git과 GitHub알면 좋음

Git: 코드 변경 내역을 저장하는 도구. 버전 관리.
GitHub: Git 코드를 인터넷에 올려두는 곳. Vercel 배포 연결도 여기서.

Git = 저장 버튼. GitHub = 클라우드. 언제든 이전 상태로 되돌릴 수 있어요.
환경변수 (API Key 보안)알면 좋음

API 키 같은 비밀 정보는 코드에 직접 넣으면 안 돼요. .env 파일에 따로 보관해요.

API 키를 코드에 넣고 GitHub에 올리면 누구나 볼 수 있어요. 이렇게 하면 키가 탈취당할 수 있어요.
Chapter 6 · 실전편
데이터가 어떻게 움직이나요?
"데이터를 어디서 가져오고, 어디에 저장하지?" 이 흐름을 알면 앱 전체 구조가 보여요.
fetch / API 호출핵심 개념

외부 서버에 데이터를 요청하는 행위예요.

const response = await fetch('https://api.example.com'); const data = await response.json();
요청(Request): "이 데이터 줘"
응답(Response): 서버가 데이터를 돌려줌
JSON: 주고받는 데이터 형식
JSON 형식꼭 알기
{ "name": "베이", "skills": ["기획", "디자인"] } // data.name → "베이" // data.skills[0] → "기획"
{ }는 객체(하나의 덩어리), [ ]는 배열(목록). 이 두 가지가 JSON의 전부예요.
동기 vs 비동기 / async·await개념 정리

API 요청은 시간이 걸려서 비동기로 처리해요. await을 붙이면 "이 작업 끝날 때까지 기다려"예요.

카페 진동벨. 커피 나올 때까지 기다리는 대신 자리에 앉아 다른 일 하다가 벨 울리면 가져오는 방식이에요.
localStorage알면 유용

서버 없이 브라우저에 데이터를 저장해요. 탭 닫아도 유지. 다른 기기에서는 안 보여요.

localStorage.setItem('key', 'value'); // 저장 localStorage.getItem('key'); // 불러오기
Chapter 7 · 심화편
구조 관련 용어 심화
React를 쓰거나 클로드가 설명할 때 자주 나오는 구조 단어들이에요.
컴포넌트 (Component)핵심 개념

화면을 이루는 독립적인 조각이에요. 헤더, 버튼, 카드 각각이 컴포넌트예요. 한 번 만들면 여러 곳에서 재사용 가능해요.

레고 블록. 블록 하나하나를 조립해서 전체 앱을 만들어요.
상태 (State)핵심 개념

앱이 지금 기억하는 값이에요. 상태가 바뀌면 화면이 자동으로 다시 그려져요(리렌더링).

신호등 색. 빨강→초록으로 바뀌면(상태 변화) 자동차가 반응(화면 변화)해요.
props (프롭스)React 용어

부모 컴포넌트가 자식에게 전달하는 설정값이에요.

<Button color="blue" text="확인" />
DOM알면 좋음

브라우저가 HTML을 읽어서 만든 구조예요. JS는 DOM을 통해 화면 요소를 읽고 바꿔요.
"DOM을 직접 조작한다" = JS로 HTML 요소를 직접 건드린다는 뜻이에요.

라이브러리 vs 프레임워크개념 정리

라이브러리: 필요한 도구를 가져다 쓰는 것. 내가 주도권을 가져요. (React, Chart.js)
프레임워크: 틀이 정해져 있고 그 안에서 코드를 채워요. (Next.js, Django)

라이브러리 = 마트에서 재료 골라 요리. 프레임워크 = 밀키트, 순서 정해짐.
Chapter 8 · 심화편
앱 종류와 기술 스택
어떤 앱을 만들 건지 정확히 표현하면 클로드가 훨씬 적합한 방향을 제안할 수 있어요.
기술 스택 (Tech Stack)필수 용어

앱을 만드는 데 쓰인 기술들의 조합이에요. "어떤 스택 쓸까요?" = 어떤 언어·프레임워크로 만들지 결정하자는 거예요.

간단한 웹앱: HTML + CSS + JS 단일 파일
React 앱: React + Tailwind CSS + Vercel
풀스택: Next.js + PostgreSQL + Vercel
정적 vs 동적 웹사이트핵심 구분

정적(Static): 내용이 고정. 포트폴리오, 소개 페이지. GitHub Pages로 배포.
동적(Dynamic): 로그인, 실시간 데이터. 서버 필요.

웹앱 vs 모바일앱 vs 데스크톱앱종류 구분

웹앱: 브라우저. 설치 불필요. 진입 장벽 가장 낮음.
모바일앱: 스마트폰 설치. React Native, Flutter.
데스크톱앱: PC 설치. Electron으로 웹 기술로도 가능.

npm / 패키지알면 좋음

npm install 패키지명 = "이 도구 설치해줘". npm은 JS 도구들의 앱스토어 같은 곳이에요.

package.json은 "이 프로젝트에 필요한 도구 목록"이에요. npm install 하면 이걸 보고 자동 설치해요.
Chapter 9 · 심화편 NEW
개발 흐름과 필수 도구들
클로드와 실제 개발할 때 흐름이 어떻게 돌아가는지, 어떤 도구들이 등장하는지 알아두면 덜 헷갈려요.
개발 → 테스트 → 배포 사이클전체 흐름
1. 로컬 개발: 내 컴퓨터에서 코드 작성 + 브라우저로 확인
2. 테스트: 기능이 제대로 동작하는지 직접 눌러보기
3. 배포: GitHub → Vercel 등으로 인터넷에 공개
4. 유지보수: 버그 수정, 기능 추가 → 1번으로
터미널 기본 명령어알면 유용
ls # 현재 폴더 파일 목록 cd 폴더명 # 폴더 이동 pwd # 지금 내 위치 npm install # 패키지 설치 npm run dev # 개발 서버 시작
클로드 코드가 대부분 대신 실행해줘요. 명령어를 외울 필요는 없고 "이런 게 있구나" 정도로만 알아도 충분해요.
리팩토링 (Refactoring)자주 나오는 단어

기능은 그대로 두고 코드를 더 깔끔하게 정리하는 작업이에요. 사용자 입장에서는 아무것도 안 바뀌어요.

클로드가 "리팩토링 해드릴까요?" = 기능 변경 없이 코드 정리. 동의하면 해달라고 해요.
빌드 (Build)자주 나오는 단어

개발용 코드를 배포용 최적화 파일로 변환하는 과정이에요. npm run build로 실행하고, Vercel은 자동으로 해줘요.

의존성 (Dependency)알면 좋음

"이 프로젝트가 동작하기 위해 필요한 외부 라이브러리들"이에요. package.json의 dependencies 항목에 적혀 있어요.

클로드가 "의존성 설치해야 해요" 하면 = npm install 실행하라는 뜻이에요.
Chapter 10 · 클로드 활용 NEW
클로드와 잘 소통하는 법
같은 요청도 어떻게 하느냐에 따라 결과가 크게 달라요. 이 패턴들을 익히면 개발 속도가 눈에 띄게 빨라져요.
클로드는 대화를 닫으면 잊어요반드시 알기

새 대화를 시작할 때는 항상 배경 설명을 먼저 해줘야 해요.

나쁜 예: "그거 고쳐줘" (뭐가 뭔지 모름)

좋은 예: "HTML+JS로 만든 할 일 앱이 있어. 완료 버튼 누르면 취소선이 생겨야 하는데 아무 반응이 없어. 코드 붙여넣을게: [코드]"
에러는 메시지 전체를 그대로 주기핵심 습관
나쁜 예: "에러 났어. 뭔가 undefined 어쩌구 하던데"

좋은 예: "이 에러 뜨는데:
TypeError: Cannot read properties of undefined (reading 'map')
at App.js:23:15"

실행 vs 배포 vs 빌드자주 혼동

실행 (run): 지금 내 컴퓨터에서 앱을 켜는 것. npm run dev
빌드 (build): 배포용 파일로 변환하는 것. npm run build
배포 (deploy): 빌드된 파일을 인터넷 서버에 올리는 것.

실행 = 집에서 요리 맛보기. 빌드 = 포장하기. 배포 = 배달 보내기.
반환(return) vs 출력(console.log)코드 읽기 팁

return: 함수가 값을 돌려주는 것. 다른 코드에서 그 값을 쓸 수 있어요.
console.log: 개발자 콘솔에 확인용으로 출력하는 것. 실제 앱 화면과는 무관해요.

클로드가 console.log를 많이 쓴다면 디버깅(문제 확인)용이에요. 배포 전에 지워도 되는 코드예요.

Q1. "클라이언트 사이드에서 처리한다"는 말의 뜻은?

클라이언트 = 브라우저(사용자 기기)예요. 서버 없이 브라우저에서만 실행된다는 뜻이에요.

Q2. const와 let의 차이는?

const(상수)는 한 번 정하면 변경 불가. let은 나중에 값을 바꿀 수 있어요.

Q3. 앱에서 빨간 에러가 떴어요. 가장 먼저 해야 할 행동은?

콘솔(F12)에서 에러 전체를 복사해서 클로드에게 주는 게 가장 빠른 해결법이에요.

Q4. Java와 JavaScript의 관계는?

Java와 JavaScript는 완전히 다른 언어예요. 이름이 비슷한 건 마케팅 이유 때문이에요.

Q5. "상태(State)가 바뀌었다"는 말의 의미는?

상태(state)는 앱이 현재 기억하는 값이에요. 상태가 바뀌면 React 등은 자동으로 화면을 다시 그려줘요(리렌더링).

Q6. "빌드(build)"란 무엇인가요?

빌드는 개발용 코드를 배포 가능한 형태로 변환하는 과정이에요. 실행(내 컴퓨터에서 켜기)과 배포(인터넷에 올리기) 사이에 있어요.

Q7. CLAUDE.md 파일의 역할은?

CLAUDE.md는 클로드 코드가 제일 먼저 읽는 지침서예요. 앱 설명, 기술 스택, 규칙을 써두면 매번 설명 안 해도 돼요.

Q8. "리팩토링(Refactoring)을 해드릴까요?"라는 클로드 제안의 의미는?

리팩토링은 기능 변경 없이 코드 구조를 개선하는 작업이에요. 사용자 입장에서는 아무것도 안 바뀌어요.

🎯 기본 개념
엣지케이스 / Edge Case

일반적이지 않은 극단적인 입력이나 상황. 정상 케이스는 잘 되는데 특정 조건에서만 터지는 버그의 원인.

예: 나이 입력칸에 0이나 999를 넣으면? / 이름에 이모지를 넣으면? / 리스트가 비어있으면?
레거시 코드 / Legacy Code

오래되어서 고치기 무서운 코드. 잘 돌아가고 있어서 건드리기 애매한 것들. "왜 이렇게 짰지?"가 자주 나옴.

클로드한테 "이 코드 레거시라 손대기 싫은데 여기만 살짝 고칠 수 있어?" 이렇게 말할 수 있어요.
리팩토링 / Refactoring

기능은 그대로인데 코드 구조를 깔끔하게 개선하는 것. 결과는 같은데 코드가 더 읽기 쉬워짐.

사용자 입장에서는 아무 변화도 없음. 개발자 입장에서는 나중에 수정하기 쉬워짐.
보일러플레이트 / Boilerplate

새 프로젝트 시작할 때 매번 반복해서 쓰는 기본 틀 코드. 내용은 거의 안 바뀌는 설정, 선언 같은 것들.

// 이런 거: 항상 파일 맨 위에 똑같이 들어가는 것들 'use strict'; import React from 'react';
하드코딩 / Hardcoding

값을 변수로 빼지 않고 코드 안에 직접 박아넣는 것. 나중에 바꾸려면 코드를 직접 열어야 해서 불편함.

// 하드코딩 (나쁜 예) if (userAge > 19) { ... } // 변수로 뺀 것 (좋은 예) const ADULT_AGE = 19; if (userAge > ADULT_AGE) { ... }
마이그레이션 / Migration

데이터나 시스템을 한 곳에서 다른 곳으로 옮기는 것. 베이 관리자도 localStorage → Firestore 마이그레이션을 했어요.

이사 개념이에요. 짐(데이터)을 새 집(새 DB)으로 옮기는 것.
기술 부채 / Tech Debt

빠르게 개발하려고 대충 짠 코드. 지금은 잘 돌아가지만 나중에 반드시 고쳐야 하는 것. 금전적 부채처럼 이자가 붙음.

클로드가 "지금은 이렇게 하고 나중에 리팩토링 하는 게 나을 것 같아요" 할 때가 기술 부채를 만드는 순간.
컨벤션 / Convention

팀이나 언어에서 암묵적으로 정한 코딩 규칙. 강제는 아니지만 안 따르면 이상하게 보임.

예: 변수명은 camelCase, 상수는 UPPER_SNAKE_CASE, 파일명은 kebab-case 등.
⚡ 함수와 비동기
콜백 / Callback

나중에 호출되도록 함수에 넘겨주는 함수. "이거 끝나면 이 함수 불러줘"의 개념.

// "파일 읽기 끝나면 이 함수(콜백)를 실행해줘" readFile('data.txt', function(result) { console.log(result); // 이게 콜백 });
비동기 / Async · Await

기다리지 않고 다음 코드 실행. 서버 응답 기다리는 동안 화면이 멈추지 않는 이유. await은 "여기서만 기다려".

// await 없이 → 데이터 오기 전에 실행됨 (버그) // await 있음 → 데이터 올 때까지 기다렸다가 실행 const data = await fetch('/api/items');
사이드 이펙트 / Side Effect

함수가 반환값 외에 외부에 영향을 주는 것. localStorage 저장, 화면 업데이트, 콘솔 출력 등. 많을수록 예측하기 어려워짐.

클로드가 "이 함수에 사이드 이펙트가 있어요"라고 하면: 이 함수가 뭔가를 바꾼다는 뜻.
클로저 / Closure

함수가 만들어질 때의 주변 변수를 기억하는 것. 함수가 사라져도 그 변수는 살아있음.

function counter() { let count = 0; // 이 변수를 return function() { count++; return count; }; // 이 함수가 기억함 }
재귀 / Recursion

함수가 자기 자신을 호출하는 것. 폴더 안에 폴더가 있는 구조나, 트리 구조 탐색에 자주 씀.

⚠️ 종료 조건이 없으면 무한 루프 → 스택 오버플로우 오류 발생
🗄️ 자료구조
스택 / Stack

마지막에 넣은 게 먼저 나오는 구조 (LIFO). 브라우저 뒤로가기, Ctrl+Z 실행취소가 이 방식.

접시를 쌓으면 맨 위(마지막 올린 것)부터 꺼내는 것과 같아요.
큐 / Queue

먼저 넣은 게 먼저 나오는 구조 (FIFO). 줄 서기랑 같음. 네트워크 요청 대기열, 알림 발송 순서에 사용.

해시맵 / HashMap (객체)

key-value 쌍의 저장 구조. JavaScript의 객체(Object)가 해시맵이에요. key로 값을 바로 꺼낼 수 있어 빠름.

const map = { 'apple': 1200, 'banana': 800 }; console.log(map['apple']); // → 1200
🌐 웹과 API
엔드포인트 / Endpoint

API에서 특정 기능을 담당하는 URL 주소. "이 주소로 요청 보내면 이 기능을 해줌".

GET /api/users ← 유저 목록 POST /api/users ← 유저 추가 DELETE /api/users/123 ← 특정 유저 삭제
페이로드 / Payload

요청이나 응답에 실제로 담기는 데이터. 택배로 치면 박스(HTTP 요청) 안에 들어있는 내용물(payload).

// POST 요청 시 body에 담기는 payload { "name": "베이", "email": "bae@example.com" }
환경변수 / Environment Variable

코드 밖에 저장하는 설정값. API 키, 비밀번호처럼 코드에 직접 쓰면 안 되는 것들을 여기에 보관.

// .env 파일 (절대 GitHub에 올리면 안 됨) FIREBASE_API_KEY=AIzaSy... DATABASE_URL=https://... // 코드에서 읽기 process.env.FIREBASE_API_KEY
웹훅 / Webhook

이벤트가 발생하면 자동으로 특정 URL을 호출해주는 것. 내가 폴링(계속 물어보기)하는 게 아니라 상대방이 알려줌.

예: GitHub에서 코드 push되면 → 자동으로 배포 서버에 알림 전송
미들웨어 / Middleware

요청과 응답 사이에 끼어서 처리하는 함수. 로그인 확인, 로그 기록, 데이터 검증 등을 요청마다 공통으로 처리할 때 씀.

공항 보안 검색대처럼, 요청이 목적지에 도달하기 전에 거쳐야 하는 관문들.
멱등성 / Idempotent

같은 요청을 여러 번 해도 결과가 같은 성질. GET, PUT은 멱등, POST는 아님 (누를 때마다 새로 생성됨).

결제 버튼을 실수로 두 번 눌러도 한 번만 결제돼야 → 멱등성이 중요한 이유
🛠️ 개발 문화와 도구
모킹 / Mocking

실제 데이터나 API 대신 가짜로 만들어서 테스트하는 것. 서버가 아직 없어도 프론트 개발 가능하게 해줌.

클로드가 "일단 목업 데이터로 개발하고 나중에 실제 API 연결할게요"라고 하면 이 뜻.
CI/CD

Continuous Integration / Continuous Deployment. 코드 올리면 자동으로 테스트하고 배포하는 파이프라인. GitHub Actions 같은 것.

베이 관리자는 main에 push하면 GitHub Pages가 자동 배포되는데, 이게 간단한 CD예요.
스코프 / Scope

변수가 살아있는 범위. 함수 안에서 만든 변수는 함수 밖에서 쓸 수 없음. 이걸 모르면 "변수가 왜 없다고 하지?" 버그가 생김.

function hello() { const name = '베이'; // 함수 스코프 안 } console.log(name); // ❌ 오류 — 밖에서 접근 불가
디펜던시 / Dependency

내 코드가 의존하는 외부 라이브러리. `npm install`로 설치하는 것들. 너무 많으면 유지보수가 어렵고, 보안 이슈가 생길 수 있음.

package.json에 적힌 것들이 모두 디펜던시예요.
스택 오버플로우 / Stack Overflow

함수 호출이 너무 많이 쌓여서 메모리가 넘치는 오류. 주로 재귀함수에서 종료 조건을 빠트렸을 때 발생.

⚠️ 사이트 stackoverflow.com과 동음이의어! 개발자들이 에러 해결법 찾는 사이트 이름이 이 오류에서 유래.
리그레션 / Regression

이전에 잘 되던 기능이 수정 후에 망가지는 것. 버그를 고쳤더니 다른 버그가 생기는 상황.

클로드가 "리그레션이 없는지 테스트해보세요"라고 하면: 기존에 잘 되던 것도 확인하세요 라는 뜻.
롤백 / Rollback

배포 후 문제가 생겨서 이전 버전으로 되돌리는 것. git revert나 이전 버전 재배포로 실행.

핫픽스 / Hotfix

배포 중인 서비스에 긴급한 버그가 생겼을 때 빠르게 고치는 패치. 다음 정기 배포를 기다릴 수 없을 때 씀.

🏗️ 자주 쓰는 구조와 패턴
플랫(Flat) vs 트리(Tree) 구조

플랫: 모든 항목이 같은 레벨. 트리: 부모-자식 계층이 있음. 오늘 만든 연결 할일이 플랫 구조예요.

// 트리: A → B → C (계층이 깊어짐) // 플랫: A → B, A → C (항상 A 아래) A ├── B ← A의 자식 └── C ← B에서 만들었지만 A의 자식
관리가 단순할 땐 플랫이 좋아요. GitHub Issues의 sub-issue, Linear의 subtask 모두 플랫 구조예요.
1:N / N:N 관계

데이터 간의 연결 방식. 1:N은 하나가 여럿을 가짐. N:N은 서로 여럿이 연결됨.

// 1:N: 카테고리 1개 → 일상 종류 여러 개 // N:N: 할일 ↔ 태그 (할일에 태그 여러 개, 태그도 여러 할일에 붙음)
베이 관리자에서: 시간표(1) : 일정(N), 카테고리(1) : 일상종류(N) 이 1:N 관계예요.
상태 머신 / State Machine

정해진 상태들과 전환 규칙의 집합. 지금 어떤 상태이고, 어떤 조건에서 다음 상태로 바뀌는지 정의함.

// 뽀모도로 상태 머신 idle → work → break → work → longbreak → ... 할일: pending → in-progress → done
클로드가 "이건 상태 머신으로 관리하는 게 좋겠어요"라고 하면: 상태가 많고 전환 규칙이 복잡할 때예요.
싱글톤 / Singleton

앱 전체에서 딱 하나만 존재하는 인스턴스. 전역 설정, DB 연결처럼 중복 생성하면 안 되는 것에 씀.

베이 관리자의 FS 객체(Firebase 연결)가 싱글톤이에요. 여러 곳에서 쓰지만 연결은 하나.
옵저버 패턴 / Observer Pattern

어떤 값이 바뀌면 구독 중인 여러 곳에 자동으로 알려주는 구조. 이벤트 리스너, Firebase의 onSnapshot이 이 패턴.

// "상태 바뀌면 알려줘" 구독 firebase.auth().onAuthStateChanged(function(user) { // 로그인 상태 바뀔 때마다 자동 호출 ← 옵저버 });
낙관적 업데이트 / Optimistic Update

서버 응답을 기다리지 않고 UI를 먼저 바꾸는 것. 빠르게 느껴지지만 실패하면 롤백해야 함.

좋아요 버튼 누르면 즉시 하트가 채워지고, 서버 실패 시 되돌아오는 것이 낙관적 업데이트예요.
단방향 데이터 흐름 / Unidirectional Data Flow

데이터가 한 방향으로만 흐르는 구조. 상태(state) → UI 렌더링 → 이벤트 → 상태 변경. React/Vue의 핵심 개념.

// 상태 → 화면 (단방향) workItems → renderWorkPage() → 화면 표시 버튼 클릭 → workItems 변경 → renderWorkPage() 재실행
SSOT / Single Source of Truth

데이터의 유일한 원본을 한 곳에만 두는 원칙. 두 곳에 같은 데이터가 있으면 동기화 문제가 생김.

베이 관리자 CLAUDE.md 첫 줄에 "SSOT"라고 되어있어요 — 프로젝트 지침의 단일 원본이라는 뜻.
이벤트 버블링 / Event Bubbling

자식 요소의 이벤트가 부모로 전파되는 것. 카드 안의 버튼 클릭이 카드 클릭도 같이 발생시키는 이유.

// 버블링 막기 button.onclick = function(e) { e.stopPropagation(); // 부모로 전파 차단 doSomething(); };
베이 관리자 코드에 onclick="event.stopPropagation()"이 많은데, 다 이 이유예요.
📝 용어 확인 퀴즈
얼마나 익혔나 확인해봐요

Q. 기능은 그대로인데 코드 구조를 깔끔하게 개선하는 것을 뭐라고 할까요?

리팩토링(Refactoring)은 결과는 동일하게 유지하면서 코드 품질을 개선하는 것이에요.

Q. 값을 변수로 빼지 않고 코드 안에 직접 박아넣는 것을 뭐라고 해요?

하드코딩(Hardcoding)이에요. 값을 상수나 환경변수로 빼는 게 좋은 습관이에요.

Q. API에서 특정 기능을 담당하는 URL 주소를 뭐라고 해요?

엔드포인트(Endpoint)예요. /api/users 같은 URL 경로 하나하나가 엔드포인트예요.

Q. 이전에 잘 되던 기능이 수정 후에 망가지는 것을 뭐라고 해요?

리그레션(Regression)이에요. 수정 전에 잘 되던 기능이 수정 후에 망가지는 것. 테스트가 중요한 이유예요.

Q. 코드 밖에 저장하는 API 키, 비밀번호 같은 설정값을 뭐라고 해요?

환경변수(Environment Variable)예요. .env 파일에 저장하고 절대 GitHub에 올리면 안 돼요!

Q. 먼저 넣은 것이 먼저 나오는 자료구조는?

큐(Queue)예요. FIFO(First In, First Out). 줄 서기랑 같아요. 스택은 반대로 LIFO(마지막에 넣은 게 먼저 나옴).

Q. Firebase의 onAuthStateChanged처럼, 값이 바뀌면 구독한 곳에 자동으로 알려주는 패턴은?

옵저버 패턴(Observer Pattern)이에요. 이벤트 리스너, onSnapshot, onAuthStateChanged 모두 이 패턴으로 동작해요.

Q. 카드 안의 버튼을 클릭했는데 카드 클릭 이벤트까지 같이 발생하는 현상은?

이벤트 버블링(Event Bubbling)이에요. e.stopPropagation()으로 막을 수 있어요. 베이 관리자 코드에도 여러 군데 쓰였어요.
🖼️ UI 레이아웃 용어 — 화면의 영역들
화면 구조 한눈에 보기꼭 알기

앱 화면은 이름 붙은 영역들로 나뉘어 있어요. 아래 그림으로 전체 구조를 먼저 잡아봐요.

Header · Navbar · Top Bar

모두 "화면 위쪽에 고정된 영역"이에요. 차이는 미묘해요.

Header (헤더): 페이지 최상단 영역. 로고, 제목, 로그인 버튼 등이 들어가요.
Navbar (내비바): Navigation Bar의 줄임. 메뉴·링크가 있는 탐색 바. Header 안에 있는 경우도 많아요.
Top Bar: 주로 모바일 앱에서 상단 고정 바를 부르는 말. 베이 관리자 모바일 상단에 있는 그거예요.
App Bar: 모바일 앱에서 Top Bar와 거의 같은 의미로 써요.

베이 관리자 코드에서 #mobileTopBar가 바로 이 Top Bar예요. 모바일에서만 보여요.
Footer · Bottom Bar · Bottom Navigation

Footer (푸터): 페이지 맨 아래 영역. 저작권, 링크, 회사 정보 등이 들어가요. 웹사이트에서 자주 보여요.

Bottom Bar: 모바일 화면 아래에 고정된 바.

Bottom Navigation (바텀 내비): 모바일 앱에서 화면 하단에 탭 아이콘들이 줄지어 있는 것. 인스타그램·유튜브 하단 메뉴가 대표적이에요.

Tab Bar: Bottom Navigation과 거의 같은 의미. iOS에서 자주 쓰는 표현이에요.

Sidebar · Drawer · Side Menu

Sidebar (사이드바): 화면 옆(주로 왼쪽)에 있는 보조 영역. 베이 관리자의 왼쪽 메뉴가 사이드바예요.

Drawer (드로어): 평소엔 숨겨져 있다가 햄버거 버튼을 누르면 옆에서 밀려나오는 메뉴. 모바일에서 사이드바를 이렇게 구현해요.

Side Menu: Drawer, Sidebar와 거의 같은 뜻으로 혼용돼요.

Modal · Dialog · Popup · Overlay자주 혼동

모두 "현재 화면 위에 떠있는 무언가"예요. 미묘하게 달라요.

Modal (모달): 화면 중앙에 뜨는 창. 뒤 배경이 어두워지고, 이걸 닫기 전까지 다른 조작이 막혀요. "삭제할까요?" 확인창이 대표적.

Dialog (다이얼로그): Modal과 거의 같은 말. "모달 다이얼로그"라고도 해요. Windows의 경고창 같은 것.

Popup (팝업): 원래는 새 브라우저 창을 여는 거지만, 요즘은 Modal과 거의 같은 뜻으로 혼용돼요.

Overlay (오버레이): 화면 위에 반투명하게 덮이는 레이어 전체를 뜻해요. Modal을 띄울 때 배경에 깔리는 어두운 부분이 오버레이예요.

┌──────────────────────────────┐ │ (Overlay — 어두운 배경) │ │ ┌────────────────────┐ │ │ │ Modal │ │ │ │ "정말 삭제할까요?" │ │ │ │ [취소] [삭제] │ │ │ └────────────────────┘ │ └──────────────────────────────┘
Bottom Sheet · Action Sheet모바일 필수

Bottom Sheet (바텀 시트): 화면 아래에서 위로 밀려 올라오는 패널이에요. 모바일 앱에서 자주 써요. 공유하기, 정렬 옵션 등을 보여줄 때 많이 사용해요.

Action Sheet (액션 시트): Bottom Sheet와 비슷한데, 주로 iOS에서 "이 중에 뭘 할까요?" 선택지를 보여줄 때 써요. 사진 앱에서 "사진 저장 / 공유 / 삭제" 같은 것.

Toast · Snackbar · Tooltip · Banner

Toast (토스트): 화면 아래(또는 위)에 잠깐 나타났다 사라지는 짧은 알림. "저장됐어요 ✓" 같은 것. 베이 관리자의 작업 완료 알림이 토스트예요.

Snackbar (스낵바): Toast와 거의 같아요. Google(Android)에서 쓰는 표현. 때로는 "실행 취소" 버튼이 달려있기도 해요.

Tooltip (툴팁): 요소에 마우스를 올리면 나오는 작은 설명 풍선. 버튼의 title 속성이 대표적.

Banner (배너): 화면 상단(또는 하단)에 길게 가로질러 표시되는 알림 바. "쿠키 동의", "앱 업데이트" 등에 많이 써요.

Toast/Snackbar: Tooltip: ┌───────────────┐ [버튼] ← 마우스 올리면 │ ✓ 저장됐어요 │ ┌──────────┐ └───────────────┘ │ 이 버튼은 │ (잠깐 떴다 사라짐) │ ~해요 │ └──────────┘
Card · Panel · Widget · Container · Box

Card (카드): 관련 정보를 묶어서 보여주는 박스. 보통 테두리나 그림자가 있어요. 베이 관리자 할일 칸반 카드, 시간표 카드가 이거예요.

Panel (패널): Card보다 큰 구역. 특정 기능 묶음을 담는 영역. 뽀모도로 패널처럼 독립된 기능 섹션에 써요.

Widget (위젯): 독립적으로 동작하는 작은 UI 컴포넌트. 날씨 위젯, 달력 위젯처럼 "그 자체로 완결된 기능 덩어리"예요.

Container (컨테이너): 다른 요소들을 담는 껍데기. 내용보다 레이아웃을 잡는 역할이에요.

Box / Wrapper (박스/래퍼): Container와 거의 같은 의미. 요소를 감싸는 역할.

FAB · Tab · Chip · Badge · Breadcrumb

FAB (Floating Action Button): 화면 위에 둥둥 떠있는 큰 원형 버튼. 베이 관리자 우측 하단의 뽀모도로 버튼, 💬 질문 저장 버튼이 FAB이에요.

Tab (탭): 여러 섹션을 전환하는 버튼. 베이 관리자 개발괴발 상단의 기초편/실전편/… 이 탭이에요.

Chip (칩): 작고 둥근 태그 모양 요소. 필터, 태그, 선택지 등에 써요. 베이 관리자 시간표 태그 칩이 이거예요.

Badge (뱃지): 숫자나 점으로 알림 개수를 표시하는 작은 표시. 카카오톡 앱 아이콘의 빨간 숫자, 개발괴발 탭의 N 표시가 뱃지예요.

Breadcrumb (브레드크럼): 현재 위치를 계층으로 보여주는 탐색 표시. 홈 > 설정 > 디자인 같은 것.

Dropdown · Menu · Popover · Context Menu

Dropdown (드롭다운): 버튼을 클릭하면 아래로 펼쳐지는 선택 목록. <select> 태그나 커스텀 선택박스가 대표적.

Menu (메뉴): 선택 가능한 항목 목록. 사이드바, 드롭다운, 컨텍스트 메뉴 등 여러 형태로 나타나요.

Popover (포오버): 요소 근처에 말풍선처럼 뜨는 작은 패널. Tooltip보다 크고 더 많은 내용을 담아요.

Context Menu (컨텍스트 메뉴): 마우스 우클릭하면 나오는 메뉴. 모바일에서는 길게 누르면 나오는 메뉴가 이에 해당해요.

✍️ WYSIWYG · 에디터 · 렌더링 용어
WYSIWYG — "보이는 게 결과물이에요"꼭 알기

WYSIWYG은 What You See Is What You Get의 줄임말이에요. 발음은 "위지윅"이에요.

편집 화면에서 보이는 그대로가 최종 결과물이 되는 방식이에요. 노션, 구글 독스, 워드처럼 굵게·기울임·색상을 적용하면 즉시 눈에 보이죠.

반대 개념은 마크다운(Markdown)이나 HTML처럼 태그/기호로 작성하고 렌더링 후에야 결과를 볼 수 있는 방식이에요.

WYSIWYG 방식: 편집창에서 → [굵게 버튼 클릭] → 즉시 굵게 보임 ✅ Markdown 방식: 편집창에서 → **굵게** 입력 → 렌더링 후 굵게 보임 (편집 중엔 별표가 그대로 보임)
노션이 WYSIWYG의 대표 예시예요. 베이 관리자 개발괴발 에디터도 일종의 WYSIWYG처럼 만들 수 있어요.
Rich Text Editor vs Plain Text Editor vs Code Editor

Plain Text Editor (플레인 텍스트 에디터): 글자만 입력하는 기본 텍스트창. 서식 없음. 메모장, <textarea>가 여기에 해당해요.

Rich Text Editor (리치 텍스트 에디터, RTE): 굵게, 밑줄, 색상, 이미지 삽입 등 서식 기능이 있는 에디터. WYSIWYG 방식이에요. 노션, 구글 독스, 네이버 카페 글쓰기창이 대표적.

Code Editor (코드 에디터): 개발자용. 문법 강조(Syntax Highlighting), 자동완성, 들여쓰기 등 코딩에 특화된 에디터. VS Code, 클로드 코드 내부 편집기가 여기에 해당해요.

Markdown (마크다운) — 가볍게 서식 쓰는 법

#, **, -, []() 같은 기호로 서식을 표현하는 경량 문법이에요. WYSIWYG과 달리 텍스트로만 작성하고, 렌더링하면 예쁘게 변환돼요.

렌더링(Rendering) — "코드를 눈에 보이는 화면으로"자주 나옴

렌더링은 코드(HTML·CSS·마크다운)를 실제 화면으로 그려내는 과정이에요.

• CSR (Client-Side Rendering): 브라우저(클라이언트)가 JavaScript를 받아서 화면을 직접 그려요. 초기 로딩이 느릴 수 있지만 이후 페이지 전환이 빨라요. 베이 관리자가 이 방식이에요.

• SSR (Server-Side Rendering): 서버가 HTML을 미리 만들어서 브라우저에 보내줘요. 초기 로딩이 빠르고 검색엔진에 유리해요. Next.js가 대표적.

• SSG (Static Site Generation): 빌드할 때 HTML을 미리 다 만들어 저장해요. 속도 최강. 블로그나 문서 사이트에 적합해요.

CSR: 브라우저 → JS 실행 → 화면 그리기 SSR: 서버 → HTML 완성 → 브라우저에 전송 SSG: 빌드 시 HTML 생성 → CDN에 저장 → 즉시 전송
GitHub Pages(베이 관리자 배포 방식)는 SSG에 가까워요. HTML 파일을 미리 올려두고 그대로 서빙해요.
CMS · Headless CMS · No-code · Low-code

CMS (Content Management System): 콘텐츠를 쉽게 관리하는 시스템. 코드 없이 글·이미지를 올리고 편집할 수 있어요. 워드프레스, 노션이 대표적이에요.

Headless CMS: 콘텐츠 관리(백엔드)만 하고, 화면 표시(프론트엔드)는 개발자가 자유롭게 구성해요. 콘텐츠를 API로 가져다 쓰는 방식.

No-code: 코드를 전혀 안 쓰고 드래그&드롭으로 앱·사이트를 만드는 방식. Webflow, Bubble이 대표적.

Low-code: 대부분은 GUI로 만들고, 필요한 부분에만 코드를 조금 써요. No-code보다 유연하지만 배움이 더 필요해요.

No-code = 레고 블록 조립 🧱 · Low-code = 레고 + 약간의 용접 🔧 · 코딩 = 처음부터 직접 제작 ⚙️
Syntax Highlighting · Live Preview · Hot Reload

Syntax Highlighting (문법 강조): 코드에서 키워드, 문자열, 함수를 색깔로 구분해 보여주는 것. VS Code에서 function은 파란색, 문자열은 초록색이 되는 그거예요.

Live Preview (라이브 프리뷰): 코드를 수정하면 결과 화면이 실시간으로 바뀌는 기능. Codepen, Figma의 실시간 미리보기가 여기에 해당해요.

Hot Reload (핫 리로드): 개발 중 코드를 저장하면 브라우저 전체를 새로고침하지 않고 변경된 부분만 자동으로 반영하는 기능. React 개발 시 파일 저장하면 화면이 알아서 바뀌는 것.

Template · Boilerplate · Scaffold

Template (템플릿): 미리 만들어둔 틀. HTML 템플릿, 이메일 템플릿처럼 기본 구조가 잡혀 있어서 내용만 채우면 돼요.

Boilerplate (보일러플레이트): 프로젝트 시작 시 매번 필요한 반복 코드 묶음. "Create React App"으로 만들어지는 기본 파일들이 보일러플레이트예요.

Scaffold (스캐폴드): 프레임워크가 자동으로 기본 파일·폴더 구조를 생성해주는 것. npx create-react-app이 스캐폴딩이에요.

Template = 빈칸 채우기 양식 📄 · Boilerplate = 항상 쓰는 기본 세팅 🔩 · Scaffold = 공사 시작 전 세우는 비계 🏗️
DOM · Virtual DOM · Shadow DOM개념

DOM (Document Object Model): 브라우저가 HTML을 읽어서 만드는 트리 구조 메모리 표현이에요. JavaScript가 document.getElementById()로 접근하는 그 구조예요.

Virtual DOM: React 등이 쓰는 기법. DOM을 직접 바꾸는 게 느려서, 가상의 복사본에서 변경사항을 계산한 뒤 실제 DOM에 한 번에 반영해요.

Shadow DOM: 컴포넌트 내부 DOM을 외부와 격리하는 기술. CSS가 밖으로 새지 않아요. 웹 컴포넌트 표준에 사용돼요.

DOM (브라우저 내부): <html> └── <body> └── <div id="app"> └── <button>클릭</button> JavaScript로 접근: document.getElementById('app') // DOM 노드를 잡음
✏️ WYSIWYG · 렌더링 퀴즈
노션처럼 편집 화면에서 보이는 그대로 최종 결과가 되는 방식은?
WYSIWYG(위지윅)이에요. What You See Is What You Get. 노션, 구글 독스가 대표적이에요.
베이 관리자(GitHub Pages)가 사용하는 렌더링 방식에 가장 가까운 것은?
베이 관리자는 미리 만든 HTML·JS 파일을 GitHub Pages에 올려두고(SSG 개념), 브라우저에서 JS가 화면을 그리는(CSR) 방식이에요.
코드를 저장하면 브라우저 전체 새로고침 없이 변경 부분만 자동 반영되는 기능은?
Hot Reload예요. React 개발 서버에서 파일 저장하면 화면이 자동으로 바뀌는 그 기능이에요.
Chapter 15 · Git 실전 NEW
브랜치 · PR · merge · diff
실제 세션에서 막혔던 부분이에요. "왜 갑자기 브랜치가 생겼지?", "PR이 뭐야?", "merge는 어떻게 해?" — 이 챕터 하나로 정리해요.
브랜치 (Branch)란?핵심 개념

코드의 복사본 작업 공간이에요. 원본(main)을 건드리지 않고 새 기능을 시험해볼 수 있어요.

문서 편집할 때 "원본은 두고 사본에서 수정해봐" 하는 것과 같아요.
# 새 브랜치 만들고 이동 git checkout -b feature/login # 이제 이 브랜치에서 작업 → main은 그대로
클로드 코드가 자동으로 브랜치를 만들기도 해요. 왜 main이 아닌 다른 브랜치인지 물어보면 이유를 설명해줘요.
main 브랜치가 특별한 이유꼭 알기

main은 "완성된, 실제 서비스에 올라가는" 브랜치예요. 여기에 직접 작업하면 실수했을 때 돌이키기 어려워요.

혼자 개발할 때: main에 직접 작업해도 사실 큰 문제는 없어요.
협업하거나 안전하게 할 때: 브랜치 → PR → merge 흐름을 써요.
merge란?핵심 개념

브랜치에서 작업한 내용을 다른 브랜치(보통 main)에 합치는 것이에요.

git checkout main # main으로 이동 git merge feature/login # feature 내용을 main에 합치기
사본에서 수정한 내용을 원본 문서에 적용하는 것이에요.
PR (Pull Request)란?핵심 개념

"내 브랜치 내용을 main에 합쳐도 될까요?"라고 요청하는 것이에요. GitHub에서 하는 작업이에요.

PR 흐름:
1. 브랜치에서 작업 완료
2. GitHub에서 PR 생성
3. diff 확인 (무엇이 바뀌었는지 검토)
4. 문제 없으면 merge 버튼 클릭
diff란?알면 좋음

변경 사항의 차이를 보여주는 것이에요. 어떤 줄이 추가됐는지(초록), 삭제됐는지(빨강).

# PR에서 이런 식으로 표시돼요 - 이 줄은 삭제됐어요 (빨간색) + 이 줄이 추가됐어요 (초록색)
push vs pull헷갈리는 쌍

push: 내 컴퓨터(로컬)의 코드를 GitHub(원격)에 올리는 것
pull: GitHub(원격)의 코드를 내 컴퓨터(로컬)로 내려받는 것

git push origin main # 로컬 → GitHub git pull origin main # GitHub → 로컬
push = 구글 드라이브에 업로드. pull = 구글 드라이브에서 다운로드.
commit이란?꼭 알기

변경 사항을 저장 + 메모하는 것이에요. 게임 세이브 포인트처럼 커밋 시점으로 언제든 돌아갈 수 있어요.

git add . # 변경된 파일 선택 git commit -m "로그인 기능 추가" # 저장 + 메모
전체 흐름 한눈에 보기정리
혼자 개발하는 가장 간단한 흐름:
코드 수정 → git add . → git commit -m "메모" → git push → 배포 자동 업데이트

브랜치를 쓰는 안전한 흐름:
새 브랜치 생성 → 코드 수정 → commit → push → GitHub에서 PR 생성 → diff 확인 → merge → main 배포
📸 실제 상황: "main에 머지할까요?" 대화창실전 독해

클로드 코드가 feature 브랜치에서 작업하다가 main에 합칠 때 이런 대화창을 띄워요. 용어를 하나씩 뜯어볼게요.

"main이 뒤처져있고 분기점만 있으므로"란?

feature 브랜치가 생긴 이후 main에 아무 커밋도 없었다는 뜻이에요.
즉, main과 feature가 같은 지점에서 출발했고 main은 그 자리에 멈춰있는 상태.
# main이 뒤처진 상황 (Fast-forward 가능한 상태) main: A → B ← 여기서 멈춤 feature: A → B → C → D → E ← 여기까지 왔음 # main이 분기한 상황 (3-way merge 필요) main: A → B → X → Y ← main도 앞으로 나감 feature: A → B → C → D ← feature도 나감 ↑ B가 분기점
Fast-forward 머지권장 방식

main이 뒤처져 있을 때, feature 커밋들을 그대로 main에 이어 붙이는 것. 합쳤다는 흔적(머지 커밋)이 없어서 히스토리가 깔끔해요.

# 전: main은 B, feature는 E까지 main: A → B feature: A → B → C → D → E # Fast-forward 머지 후: main이 E까지 쭉 이어짐 main: A → B → C → D → E ✓ 깔끔!
혼자 작업할 땐 거의 항상 이 방식이 좋아요. "충돌 없이 깔끔하게 머지"가 되는 조건이기도 해요.
PR 만들어서 머지검토 포함

GitHub에서 PR(Pull Request)을 생성하고, diff를 확인한 뒤 머지 버튼을 누르는 방식. 한 단계가 더 있지만 변경 내용을 눈으로 확인할 수 있어요.

혼자 개발: PR 없이 Fast-forward가 더 빠름
팀 개발: PR이 필수 — 다른 사람이 코드 리뷰 후 머지
cherry-pick선택적 가져오기

브랜치 전체를 머지하는 게 아니라 특정 커밋만 골라서 다른 브랜치에 가져오는 것.

# feature에 커밋이 3개 있는데 그 중 하나만 main에 가져오고 싶을 때 git cherry-pick abc1234 # 커밋 해시 지정
버그 수정 커밋만 급하게 main에 반영해야 할 때 씁니다. 체리를 하나씩 골라 담는 것처럼요.
커밋 해시 (Commit Hash)알면 좋음

각 커밋에 자동으로 붙는 고유 ID. git log를 보면 옆에 긴 영숫자가 붙어요. cherry-pick 같은 명령에서 "어떤 커밋인지" 특정할 때 씁니다.

git log --oneline abc1234 로그인 기능 추가 ← 여기서 abc1234가 해시 def5678 버그 수정 ghi9012 초기 설정
머지 충돌 (Merge Conflict)꼭 알기

두 브랜치가 같은 파일의 같은 줄을 다르게 수정했을 때 발생. Git이 "어떤 걸 쓸지 나는 모르겠으니 직접 골라"라고 멈추는 상황.

# 충돌이 나면 파일 안에 이런 표시가 생겨요 <<<<<<< HEAD (현재 브랜치) 이쪽 내용 ======= 저쪽 내용 >>>>>>> feature/login (합치려는 브랜치)
⚠️ 충돌이 나면 파일을 직접 열어서 원하는 내용만 남기고 나머지 표시를 지운 뒤, 다시 커밋해야 해요.
Chapter 16 · 개발 환경 실전 NEW
작업하다 갑자기 튀어나오는 개념들
클로드 코드랑 실제로 개발하다 보면 "이게 뭐지?" 싶은 단어들이 갑자기 등장해요. 브랜치·PR 정도 수준의 것들만 골라서 정리했어요.
HEAD자주 보임

지금 내가 작업하고 있는 현재 위치예요. "로컬 HEAD ↔ origin 동기화 완료"는 내 컴퓨터와 GitHub가 같은 상태라는 뜻이에요.

대부분 신경 안 써도 돼요. 클로드가 "HEAD가 어쩌구" 하면 그냥 "현재 위치" 정도로 이해하면 충분해요.
origin자주 보임

GitHub에 있는 내 레포지토리(원격 저장소)의 별명이에요. git push origin main은 "origin(GitHub)의 main 브랜치에 올려줘"라는 뜻이에요.

git push origin main # origin = GitHub 주소의 별명 git pull origin main # GitHub에서 내려받기
충돌 (Conflict)당황하지 말기

merge할 때 같은 줄을 두 곳에서 다르게 수정했을 때 발생해요. Git이 "어느 걸 써야 해요?"라고 묻는 상황이에요.

충돌이 나면 파일 안에 이런 표시가 생겨요:

<<<<<<< HEAD
내 버전의 코드
=======
상대방의 코드
>>>>>>> feature/login
충돌 났을 때 클로드에게 "충돌 났어, 파일 보여줄게"라고 하면 해결해줘요.
스테이징 영역 (git add가 왜 따로 있냐)궁금했을 것

git add는 "이 파일을 다음 커밋에 포함할게"라고 선택하는 단계예요. 수정한 파일이 10개여도 그 중 3개만 골라서 커밋할 수 있어요.

택배 포장. 물건을 박스에 넣고(add) 테이프로 봉하는(commit) 두 단계예요.
포트 (Port) — localhost:3000의 3000자주 보임

같은 컴퓨터에서 여러 서버를 동시에 실행하기 위한 번호예요.

localhost:3000 → 3000번 포트에서 실행 중인 앱
localhost:8080 → 8080번 포트에서 실행 중인 앱
"포트 3000이 이미 사용 중" 에러 = 이미 그 번호로 다른 앱이 실행되고 있다는 뜻이에요.
환경 (Environment) — dev / staging / production꼭 알기
development (dev): 개발 중인 환경. 내 컴퓨터. 막 고쳐도 됨.
staging: 배포 전 테스트 환경. 실제랑 똑같이 만들어둔 테스트 공간.
production (prod): 실제 사용자가 쓰는 환경. 신중하게 다뤄야 함.
CORS 에러자주 만남

브라우저가 다른 주소의 서버에 요청할 때 막히는 보안 정책이에요.

CORS 에러 나면 클로드에게 에러 메시지 그대로 붙여넣으면 돼요.
캐시 (Cache)알면 유용

속도를 위해 데이터를 미리 저장해두는 것이에요. "수정했는데 변경이 안 보여요" 현상의 원인이 되기도 해요.

브라우저 캐시 지우기: Ctrl+Shift+R (강력 새로고침).
토큰 (Token) vs API 키알면 좋음

API 키: 서비스에 접근하는 영구적인 비밀번호 같은 것.
토큰(Token): 로그인 후 임시로 발급되는 출입증. 보통 시간이 지나면 만료돼요.

.env 파일 vs Secrets꼭 구분하기

.env 파일: 내 컴퓨터(로컬)에서 쓰는 비밀 설정 파일. GitHub에 올리면 안 돼요.
Secrets (GitHub / Vercel): 배포 환경에서 쓰는 비밀 설정.

.gitignore 파일에 .env가 들어 있어야 GitHub에 안 올라가요.
CRUD꼭 알기

데이터를 다루는 4가지 기본 동작의 앞글자를 딴 말이에요.

Create — 만들기 (게시글 쓰기, 회원가입, 할 일 추가)
Read — 읽기 (목록 불러오기, 상세 페이지 보기)
Update — 수정하기 (게시글 편집, 프로필 변경)
Delete — 삭제하기 (게시글 지우기, 계정 탈퇴)

브랜치 실전 생존 가이드
브랜치 때문에 작업 내용이 유실되거나, 어느 브랜치에 있는지 몰라 혼란스러웠던 경험 — 이 챕터 하나로 정리해요.
지금 내가 어느 브랜치에 있는지 항상 확인하기가장 중요

작업 시작 전에 현재 브랜치를 확인하는 습관이 가장 중요해요. 모르고 엉뚱한 브랜치에서 작업하면 나중에 꼬여요.

git branch # 브랜치 목록 + 현재 위치(* 표시) git status # 현재 브랜치 + 변경된 파일 목록
클로드 코드가 브랜치를 자동으로 만드는 이유꼭 알기

클로드 코드는 작업할 때 자동으로 새 브랜치를 만들기도 해요. main을 보호하고 작업 단위를 분리하기 위해서예요.

클로드 코드가 새 브랜치를 만들었다면 작업 후 반드시 "이 브랜치 main에 merge해줘"라고 말해야 해요.
작업 내용이 사라진 것처럼 보일 때유실 사고 원인 1

브랜치를 이동하면 파일 내용이 그 브랜치 상태로 바뀌어요. 사라진 게 아니라 다른 브랜치에 있는 거예요.

# 내가 작업하던 브랜치로 다시 이동 git checkout feature/내가-작업하던-브랜치 # 브랜치 목록 모를 때 git branch
커밋 안 하고 브랜치 이동하면 생기는 일유실 사고 원인 2
안전한 브랜치 이동 순서:
1. 작업 완료
2. git add . → git commit -m "작업 내용" (저장 먼저)
3. 그 다음 git checkout 다른브랜치
커밋하지 않고 브랜치를 이동하는 건 위험해요. 이동하기 전에 항상 커밋하거나, stash를 쓰세요.
stash — 임시 저장유용한 도구

커밋하기엔 애매한데 일단 작업 내용을 잠깐 치워둘 때 쓰는 임시 보관함이에요.

git stash # 현재 변경사항 임시 저장 (작업 공간 깨끗해짐) git stash pop # 임시 저장한 내용 다시 꺼내기
서랍에 잠깐 넣어두기. 나중에 꺼내면 그대로 있어요.
reset — 가장 위험한 명령어위험
git reset --soft: 커밋만 취소, 파일 변경사항은 남아있음 (비교적 안전)

git reset --mixed: 커밋 취소 + add도 취소, 파일은 남아있음

git reset --hard: 커밋 취소 + 파일 변경사항까지 전부 삭제 (위험!)
클로드 코드가 reset을 하려고 할 때 어떤 종류인지, 어떤 커밋으로 돌아가는지 꼭 물어봐요.
작업 흐름 — 클로드 코드와 안전하게 쓰는 패턴추천 루틴
매 작업 시작 전
→ "지금 브랜치 상태 확인해줘" (현재 위치 파악)

기능 하나 만들 때
→ "새 브랜치 만들어서 [기능] 작업 시작해줘"

작업 중간중간
→ "지금까지 작업 커밋해줘" (중간 저장)

기능 완성됐을 때
→ "main에 merge하고 브랜치 정리해줘"

세션 끝내기 전
→ "커밋 안 된 내용 있으면 커밋하고 push까지 해줘"
세션을 그냥 닫기 전에 반드시 커밋·push 여부를 확인해요. 이게 유실 사고의 거의 모든 원인이에요.
이 상황 전체를 베이 언어로 정리스크린샷 해석
"브랜치 분기 문제가 있네요. merge로 해결하고, tool로 진행할게요."
→ 브랜치가 꼬여있어서 먼저 합치는 작업(merge)을 하고, 그 다음에 파일을 직접 수정할게요.

"워킹 트리가 깨끗하네요."
→ 커밋 안 된 변경사항은 없어요. 일단 안전한 상태예요.

"로컬 main이 origin/main과 완전히 다른 히스토리를 가지고 있어요"
→ 내 컴퓨터의 코드 기록이랑 GitHub의 코드 기록이 서로 달라요.

"로컬을 origin/main으로 리셋할게요."
→ GitHub 기준으로 내 컴퓨터를 강제로 맞출게요. ⚠️ 로컬에만 있던 작업 내용은 사라져요.

클로드 코드 웹 환경 이해하기
Remote(클라우드) vs Local(로컬) — 클로드가 돌아가는 곳꼭 알기

클로드 코드를 어디서 여냐에 따라 실행 환경이 완전히 달라요.

Remote(클라우드) ☁️: claude.ai/code에서 열면 클로드가 클라우드 서버(인터넷 어딘가에 있는 컴퓨터)에서 실행돼요. 지금 이 베이 관리자 개발이 바로 이 방식이에요.

Local(로컬) 💻: 내 Mac 터미널에서 claude 명령어로 열면 클로드가 내 컴퓨터 안에서 실행돼요. 내 Git 계정, 내 비밀번호, 내 파일 등을 직접 사용할 수 있어요.

핵심 차이: Remote 환경에서는 클로드가 내 GitHub 계정에 직접 접근할 권한이 없어요. 그래서 push 방식이나 브랜치 전략이 달라지는 거예요.

"스크린샷에 Remote(☁️) 표시가 있어요" → 지금 클라우드 서버에서 클로드가 실행 중이라는 뜻이에요.
feature 브랜치 자동 생성 (claude/이름-랜덤코드)꼭 알기

웹(Remote) 환경에서 클로드는 main 브랜치에 직접 작업하지 않아요. 대신 매 세션마다 임시 브랜치를 자동으로 만들어요.

형식: claude/작업이름-랜덤코드
예: claude/setup-environment-detection-MYc81

왜 이렇게 하냐면?
main 브랜치는 실제 서비스에 배포되는 중요한 브랜치예요. 작업 중인 미완성 코드가 바로 main에 들어가면 앱이 깨질 수 있어요. 그래서 별도 브랜치에서 작업하고, 완성되면 검토 후 합치는 방식을 써요.

# 현재 브랜치 확인 git branch # * claude/setup-environment-... 처럼 뜨면 feature 브랜치 git branch # * main 이면 main 브랜치
git push 403 에러 — "웹 환경에서 main에 직접 push 안 돼요"자주 만남

git push origin main을 실행했을 때 403 에러가 뜨는 건 "권한이 없어요"라는 뜻이에요.

웹(Remote) 환경에서는 클로드가 GitHub 계정을 직접 갖고 있지 않아서 main 브랜치에 바로 push할 권한이 없어요.

해결 흐름:
1️⃣ feature 브랜치에 push (이건 가능해요)
2️⃣ PR(Pull Request) 생성 — "이 브랜치를 main에 합쳐도 될까요?" 요청
3️⃣ 검토 후 머지(merge) — feature 브랜치가 main에 합쳐짐
4️⃣ GitHub Pages가 main 기준으로 자동 배포

베이 관리자에서는 push.sh 스크립트를 써서 이 문제를 우회하고 있어요. 클로드가 알아서 처리하니 신경 안 써도 돼요.
MCP 도구 (mcp__github__ 등) — 클로드의 손발개념

MCP(Model Context Protocol)는 클로드가 외부 서비스를 직접 다룰 수 있게 해주는 연결 규격이에요.

mcp__github__: GitHub에서 PR 만들기, 머지하기, 이슈 보기 등을 클로드가 직접 실행
mcp__notion__: Notion 페이지 읽기/쓰기
mcp__firebase__: Firestore 데이터 읽기/쓰기 (연결하면 저장된 질문 바로 확인 가능!)

MCP가 없으면 클로드는 코드 파일만 수정할 수 있어요. MCP가 있으면 GitHub, Notion, Firebase 같은 외부 서비스까지 직접 조작할 수 있어요.

설정 방법: ~/.claude/settings.json에 MCP 서버를 등록하면, 다음 세션부터 해당 도구가 활성화돼요.

클로드가 대화 중에 mcp__github__create_pull_request 같은 걸 쓴다면, GitHub MCP 도구로 PR을 만들고 있다는 뜻이에요.

.bat 파일이란? — 서버 기반 웹앱의 시작점

.bat(배치 파일, Batch File)은 Windows에서 명령어 여러 개를 자동으로 실행해주는 파일이에요.

서버 기반 웹앱은 브라우저 외에 내 컴퓨터에서 서버가 먼저 켜져 있어야 해요. .bat는 그 서버를 자동으로 시작해주는 역할이에요.

# start.bat (Node.js 서버 시작) @echo off npm install # 필요한 라이브러리 설치 npm run dev # 개발 서버 실행 pause 실행하면: Node.js 실행 → 웹 서버 시작 → localhost:3000 열림 → 브라우저에서 웹앱 접속 # 또는 Python 서버 python app.py # Flask 서버 시작 → localhost:5000
서버 기반 프로젝트 구조: 프로젝트/ ├── install.bat # 라이브러리 설치용 ├── start.bat # 서버 시작용 ├── package.json # 라이브러리 목록 ├── src/ # 소스코드 └── public/ # 정적 파일
