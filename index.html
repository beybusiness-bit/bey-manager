Chapter 1 · 기초편
웹 앱은 어떻게 생겼나요?
웹 앱을 만들기 전에 전체 구조를 이해해두면 클로드에게 요청하고 설명하기가 훨씬 쉬워져요.
프론트엔드 (화면)꼭 알기

사용자 눈에 보이는 모든 것이에요. 버튼, 텍스트, 이미지, 입력창. HTML/CSS/JavaScript로 만들어지고 사용자의 브라우저에서 실행돼요.

식당의 홀 — 손님이 직접 보고 앉고 메뉴를 고르는 공간이에요.
백엔드 (서버)알면 좋음

사용자 눈에 안 보이지만 실제 계산·저장·로그인 처리가 일어나는 곳이에요. Python, Node.js 같은 언어로 만들고 서버 컴퓨터에서 실행돼요.

식당의 주방 — 손님은 못 들어가지만 음식이 만들어지는 곳이에요.
데이터베이스 (DB)알면 좋음

데이터를 저장하는 창고예요. 회원 정보, 게시글, 주문 내역 등이 여기에 저장돼요. 간단한 앱은 Google Sheets를 DB 대용으로 쓰기도 해요.

식당의 냉장고 — 재료(데이터)를 보관해두고 필요할 때 꺼내요.
클라이언트 vs 서버꼭 알기

클라이언트는 요청하는 쪽(브라우저, 앱)이고, 서버는 응답하는 쪽이에요. "클라이언트 사이드 처리" = 서버 없이 브라우저에서만 실행된다는 뜻이에요.

클로드가 "클라이언트에서 처리해요" 하면 = 내 브라우저에서만 동작하는 코드라는 뜻이에요. 서버 비용이 안 들어요.
Chapter 2 · 기초편
개발에서 자주 쓰는 언어들
언어 이름만 들어도 막막하죠. 각각 어떤 역할인지만 알면 충분해요.
HTML — 뼈대꼭 알기

"여기 제목", "여기 버튼", "여기 이미지" 구조를 잡아요. 꺾쇠(< >)로 감싸는 태그를 씁니다.

<!-- 주석: 이런 식으로 생겼어요 --> <h1>안녕하세요</h1> <button>클릭하기</button>
CSS — 디자인꼭 알기

색, 폰트, 크기, 여백, 버튼 모양을 정해요. HTML이 뼈대라면 CSS는 옷이에요.

button { background-color: blue; border-radius: 8px; }
JavaScript (JS) — 동작꼭 알기

버튼 클릭 반응, 데이터 계산, 서버 통신이 JS로 이루어져요. 세 언어 중 가장 복잡하지만 클로드가 전부 써줘요.

"JS", "자바스크립트", "스크립트"는 모두 같은 말이에요. Java와는 완전히 다른 언어예요 — 이름만 비슷해요!
TypeScript (TS)나중에

JavaScript에 타입 체크 기능을 추가한 언어예요. 클로드가 TS로 코드를 쓰기도 해요. "JS랑 거의 같은데 더 엄격한 것"으로 이해하면 충분해요.

클로드가 .ts 파일을 만들었다면 TypeScript예요. 당황하지 말아요 — JS처럼 쓰면 돼요.
Chapter 3 · 실전편
클로드가 준 코드, 어떻게 읽나요?
직접 짤 줄 몰라도 구조를 눈으로 읽을 수 있으면 대화가 훨씬 쉬워져요.
함수 (function)핵심 개념

어떤 작업을 묶어서 이름을 붙인 것이에요. ()가 붙어 있으면 함수 호출이에요.

function greet(name) { return "안녕, " + name; } greet("베이"); // → "안녕, 베이"
레시피 카드. 이름을 붙여두고 필요할 때마다 꺼내 써요.
변수 (variable)핵심 개념

값을 담아두는 이름표예요. const는 변경 불가, let은 변경 가능.

const appName = "베이 앱"; // 안 바뀜 let count = 0; // 바뀔 수 있음 count = count + 1; // 이제 1
조건문 (if)핵심 개념

"만약 ~이면 이렇게"를 표현해요. 거의 모든 코드에 등장해요.

if (isLoggedIn) { showDashboard(); } else { showLoginPage(); }
화살표 함수 (Arrow Function)자주 보임

함수를 짧게 쓰는 문법이에요. 클로드가 자주 씁니다.

// 일반 함수와 같은 동작이에요 const add = (a, b) => a + b; // 또는 items.map(item => item.name)
=> 가 보이면 "아, 함수구나"라고 인식하면 돼요.
Chapter 4 · 실전편
에러 메시지 해석하기
빨간 글씨가 떠도 당황하지 말아요. 패턴만 알면 겁이 훨씬 덜 나요.
에러 메시지 구조핵심 패턴
에러 종류: 뭐가 잘못됐는지 (예: TypeError)
에러 내용: 구체적으로 어떤 문제인지
위치: 몇 번째 줄에서 났는지 (at line 23)
전부 이해 안 해도 돼요. 에러 메시지 전체를 복사해서 클로드에게 주면 돼요.
자주 보는 에러 종류참고용

is not defined → 없는 변수·함수를 썼어요
Cannot read properties of undefined → 값이 없는데 뭔가 꺼내려 했어요
SyntaxError → 괄호가 안 닫혔거나 오타
404 Not Found → 요청한 주소·파일이 없어요
401 Unauthorized → 로그인 필요 or 권한 없음
500 Internal Server Error → 서버 쪽 문제

콘솔(Console) 여는 법꼭 알기

Windows/Linux: F12 → Console 탭
Mac: Cmd + Option + J

앱이 이상하면 콘솔부터 열어요. 빨간 글씨 = 에러, 노란 글씨 = 경고예요.
"빌드 실패" vs "런타임 에러"구분 팁

빌드 에러: 앱을 시작하기 전에 나는 에러. 문법 오류가 많아요. 터미널에 떠요.
런타임 에러: 앱이 실행 중에 나는 에러. 특정 동작을 할 때 터져요. 콘솔에 떠요.

터미널이 빨개서 앱이 아예 안 켜진다 = 빌드 에러. 앱은 켜졌는데 버튼 누를 때 터진다 = 런타임 에러.
Chapter 5 · 실전편
파일, 폴더 구조와 배포
프로젝트 파일 구조를 알면 클로드가 뭘 만들고 있는지 훨씬 쉽게 파악돼요.
기본 웹앱 파일 구조꼭 알기
my-app/ ├── index.html # 시작 페이지 ├── style.css # 디자인 ├── script.js # 기능 └── images/ # 이미지 폴더
단일 HTML 파일 앱은 CSS·JS를 HTML 안에 전부 넣기도 해요. 간단한 앱에 많이 쓰이는 방식이에요.
로컬 vs 배포꼭 알기

로컬(local): 내 컴퓨터에서만 돌아가요. 주소가 localhost:3000처럼 생겼어요.

배포(deploy): 인터넷 서버에 올려서 누구나 접속 가능. 실제 URL이 생겨요.

"로컬에서는 되는데 배포 후에 안 돼요"는 매우 흔한 상황이에요. 환경이 달라서 생기는 문제예요.
무료 배포 도구꼭 알기

GitHub Pages: HTML 파일 무료 배포. 정적 사이트에 딱이에요.
Vercel / Netlify: React 같은 복잡한 앱도 무료 배포 가능. 더 추천.
Replit: 코딩 + 배포를 한 곳에서.

Git과 GitHub알면 좋음

Git: 코드 변경 내역을 저장하는 도구. 버전 관리.
GitHub: Git 코드를 인터넷에 올려두는 곳. Vercel 배포 연결도 여기서.

Git = 저장 버튼. GitHub = 클라우드. 언제든 이전 상태로 되돌릴 수 있어요.
환경변수 (API Key 보안)알면 좋음

API 키 같은 비밀 정보는 코드에 직접 넣으면 안 돼요. .env 파일에 따로 보관해요.

API 키를 코드에 넣고 GitHub에 올리면 누구나 볼 수 있어요. 이렇게 하면 키가 탈취당할 수 있어요.
Chapter 6 · 실전편
데이터가 어떻게 움직이나요?
"데이터를 어디서 가져오고, 어디에 저장하지?" 이 흐름을 알면 앱 전체 구조가 보여요.
fetch / API 호출핵심 개념

외부 서버에 데이터를 요청하는 행위예요.

const response = await fetch('https://api.example.com'); const data = await response.json();
요청(Request): "이 데이터 줘"
응답(Response): 서버가 데이터를 돌려줌
JSON: 주고받는 데이터 형식
JSON 형식꼭 알기
{ "name": "베이", "skills": ["기획", "디자인"] } // data.name → "베이" // data.skills[0] → "기획"
{ }는 객체(하나의 덩어리), [ ]는 배열(목록). 이 두 가지가 JSON의 전부예요.
동기 vs 비동기 / async·await개념 정리

API 요청은 시간이 걸려서 비동기로 처리해요. await을 붙이면 "이 작업 끝날 때까지 기다려"예요.

카페 진동벨. 커피 나올 때까지 기다리는 대신 자리에 앉아 다른 일 하다가 벨 울리면 가져오는 방식이에요.
localStorage알면 유용

서버 없이 브라우저에 데이터를 저장해요. 탭 닫아도 유지. 다른 기기에서는 안 보여요.

localStorage.setItem('key', 'value'); // 저장 localStorage.getItem('key'); // 불러오기
Chapter 7 · 심화편
구조 관련 용어 심화
React를 쓰거나 클로드가 설명할 때 자주 나오는 구조 단어들이에요.
컴포넌트 (Component)핵심 개념

화면을 이루는 독립적인 조각이에요. 헤더, 버튼, 카드 각각이 컴포넌트예요. 한 번 만들면 여러 곳에서 재사용 가능해요.

레고 블록. 블록 하나하나를 조립해서 전체 앱을 만들어요.
상태 (State)핵심 개념

앱이 지금 기억하는 값이에요. 상태가 바뀌면 화면이 자동으로 다시 그려져요(리렌더링).

신호등 색. 빨강→초록으로 바뀌면(상태 변화) 자동차가 반응(화면 변화)해요.
props (프롭스)React 용어

부모 컴포넌트가 자식에게 전달하는 설정값이에요.

<Button color="blue" text="확인" />
DOM알면 좋음

브라우저가 HTML을 읽어서 만든 구조예요. JS는 DOM을 통해 화면 요소를 읽고 바꿔요.
"DOM을 직접 조작한다" = JS로 HTML 요소를 직접 건드린다는 뜻이에요.

라이브러리 vs 프레임워크개념 정리

라이브러리: 필요한 도구를 가져다 쓰는 것. 내가 주도권을 가져요. (React, Chart.js)
프레임워크: 틀이 정해져 있고 그 안에서 코드를 채워요. (Next.js, Django)

라이브러리 = 마트에서 재료 골라 요리. 프레임워크 = 밀키트, 순서 정해짐.
Chapter 8 · 심화편
앱 종류와 기술 스택
어떤 앱을 만들 건지 정확히 표현하면 클로드가 훨씬 적합한 방향을 제안할 수 있어요.
기술 스택 (Tech Stack)필수 용어

앱을 만드는 데 쓰인 기술들의 조합이에요. "어떤 스택 쓸까요?" = 어떤 언어·프레임워크로 만들지 결정하자는 거예요.

간단한 웹앱: HTML + CSS + JS 단일 파일
React 앱: React + Tailwind CSS + Vercel
풀스택: Next.js + PostgreSQL + Vercel
정적 vs 동적 웹사이트핵심 구분

정적(Static): 내용이 고정. 포트폴리오, 소개 페이지. GitHub Pages로 배포.
동적(Dynamic): 로그인, 실시간 데이터. 서버 필요.

웹앱 vs 모바일앱 vs 데스크톱앱종류 구분

웹앱: 브라우저. 설치 불필요. 진입 장벽 가장 낮음.
모바일앱: 스마트폰 설치. React Native, Flutter.
데스크톱앱: PC 설치. Electron으로 웹 기술로도 가능.

npm / 패키지알면 좋음

npm install 패키지명 = "이 도구 설치해줘". npm은 JS 도구들의 앱스토어 같은 곳이에요.

package.json은 "이 프로젝트에 필요한 도구 목록"이에요. npm install 하면 이걸 보고 자동 설치해요.
Chapter 9 · 심화편 NEW
개발 흐름과 필수 도구들
클로드와 실제 개발할 때 흐름이 어떻게 돌아가는지, 어떤 도구들이 등장하는지 알아두면 덜 헷갈려요.
개발 → 테스트 → 배포 사이클전체 흐름
1. 로컬 개발: 내 컴퓨터에서 코드 작성 + 브라우저로 확인
2. 테스트: 기능이 제대로 동작하는지 직접 눌러보기
3. 배포: GitHub → Vercel 등으로 인터넷에 공개
4. 유지보수: 버그 수정, 기능 추가 → 1번으로
터미널 기본 명령어알면 유용
ls # 현재 폴더 파일 목록 cd 폴더명 # 폴더 이동 pwd # 지금 내 위치 npm install # 패키지 설치 npm run dev # 개발 서버 시작
클로드 코드가 대부분 대신 실행해줘요. 명령어를 외울 필요는 없고 "이런 게 있구나" 정도로만 알아도 충분해요.
리팩토링 (Refactoring)자주 나오는 단어

기능은 그대로 두고 코드를 더 깔끔하게 정리하는 작업이에요. 사용자 입장에서는 아무것도 안 바뀌어요.

클로드가 "리팩토링 해드릴까요?" = 기능 변경 없이 코드 정리. 동의하면 해달라고 해요.
빌드 (Build)자주 나오는 단어

개발용 코드를 배포용 최적화 파일로 변환하는 과정이에요. npm run build로 실행하고, Vercel은 자동으로 해줘요.

의존성 (Dependency)알면 좋음

"이 프로젝트가 동작하기 위해 필요한 외부 라이브러리들"이에요. package.json의 dependencies 항목에 적혀 있어요.

클로드가 "의존성 설치해야 해요" 하면 = npm install 실행하라는 뜻이에요.

claude.ai 채팅 (지금 여기)

CLAUDE.md 파일핵심 도구

CLAUDE.md에 써두면 좋은 것:
- 이 앱이 어떤 앱인지 (목적, 사용자)
- 기술 스택 (React? 순수 HTML?)
- 코딩 스타일 규칙 (한국어 주석, 탭 크기 등)
- 하면 안 되는 것 (예: 외부 CSS 프레임워크 쓰지 말기)
클로드 코드 요청 패턴실전 팁
기능 추가: "현재 앱에 검색 기능 추가해줘. 입력창에 치면 목록이 필터링되게"

버그 수정: "버튼 클릭 시 페이지가 새로고침돼. 막아줘"

전체 제작: "사용자가 메모 작성·저장할 수 있는 웹앱. localStorage 사용"
클로드 코드가 파일 여러 개를 만들 때상황 대처

클로드 코드가 한 번에 파일을 여러 개 만들고 수정할 때, 어떤 파일을 왜 만들었는지 물어봐도 돼요.

"방금 만든 파일들 각각 뭐 하는 건지 설명해줘" — 이렇게 물어보면 각 파일의 역할을 설명해줘요.

Java vs JavaScript완전히 다름

이름만 비슷하고 완전히 다른 언어예요. Java는 주로 기업용 백엔드나 안드로이드 앱, JavaScript는 웹 프론트엔드에 써요. 혼동하는 사람이 정말 많아요.

웹 개발 맥락에서 "스크립트", "JS" = JavaScript예요. Java를 쓸 일은 거의 없어요.
라이브러리 vs 프레임워크 (다시)자주 혼동

둘 다 "남이 만들어둔 코드 모음"이지만 주도권이 달라요.

라이브러리: 내가 필요할 때 불러 써요. (React, Lodash)
프레임워크: 프레임워크 규칙에 맞게 내 코드를 끼워 넣어요. (Next.js, Vue)

서버 vs 서버리스 (Serverless)자주 혼동

서버: 24시간 돌아가는 컴퓨터.
서버리스: 서버가 없는 게 아니라, 내가 직접 서버를 관리 안 해도 되는 방식이에요. Vercel이 대표적이에요.

Vercel에 배포하면 "서버리스 배포"예요. 서버가 없는 게 아니라 Vercel이 대신 관리해줘요.
실행 vs 배포 vs 빌드자주 혼동

실행 (run): 지금 내 컴퓨터에서 앱을 켜는 것. npm run dev
빌드 (build): 배포용 파일로 변환하는 것. npm run build
배포 (deploy): 빌드된 파일을 인터넷 서버에 올리는 것.

실행 = 집에서 요리 맛보기. 빌드 = 포장하기. 배포 = 배달 보내기.
반환(return) vs 출력(console.log)코드 읽기 팁

return: 함수가 값을 돌려주는 것. 다른 코드에서 그 값을 쓸 수 있어요.
console.log: 개발자 콘솔에 확인용으로 출력하는 것. 실제 앱 화면과는 무관해요.

클로드가 console.log를 많이 쓴다면 디버깅(문제 확인)용이에요. 배포 전에 지워도 되는 코드예요.

맞힌 문제:0/ 0

Q1. "클라이언트 사이드에서 처리한다"는 말의 뜻은?

클라이언트 = 브라우저(사용자 기기)예요. 서버 없이 브라우저에서만 실행된다는 뜻이에요.

Q2. const와 let의 차이는?

const(상수)는 한 번 정하면 변경 불가. let은 나중에 값을 바꿀 수 있어요.

Q3. 앱에서 빨간 에러가 떴어요. 가장 먼저 해야 할 행동은?

콘솔(F12)에서 에러 전체를 복사해서 클로드에게 주는 게 가장 빠른 해결법이에요.

Q4. Java와 JavaScript의 관계는?

Java와 JavaScript는 완전히 다른 언어예요. 이름이 비슷한 건 마케팅 이유 때문이에요.

Q5. "상태(State)가 바뀌었다"는 말의 의미는?

상태(state)는 앱이 현재 기억하는 값이에요. 상태가 바뀌면 React 등은 자동으로 화면을 다시 그려줘요(리렌더링).

Q6. "빌드(build)"란 무엇인가요?

빌드는 개발용 코드를 배포 가능한 형태로 변환하는 과정이에요. 실행(내 컴퓨터에서 켜기)과 배포(인터넷에 올리기) 사이에 있어요.

Q7. CLAUDE.md 파일의 역할은?

CLAUDE.md는 클로드 코드가 제일 먼저 읽는 지침서예요. 앱 설명, 기술 스택, 규칙을 써두면 매번 설명 안 해도 돼요.

Q8. "리팩토링(Refactoring)을 해드릴까요?"라는 클로드 제안의 의미는?

리팩토링은 기능 변경 없이 코드 구조를 개선하는 작업이에요. 사용자 입장에서는 아무것도 안 바뀌어요.

Chapter 15 · Git 실전 NEW
브랜치 · PR · merge · diff
실제 세션에서 막혔던 부분이에요. "왜 갑자기 브랜치가 생겼지?", "PR이 뭐야?", "merge는 어떻게 해?" — 이 챕터 하나로 정리해요.
브랜치 (Branch)란?핵심 개념

코드의 복사본 작업 공간이에요. 원본(main)을 건드리지 않고 새 기능을 시험해볼 수 있어요.

문서 편집할 때 "원본은 두고 사본에서 수정해봐" 하는 것과 같아요.
# 새 브랜치 만들고 이동 git checkout -b feature/login # 이제 이 브랜치에서 작업 → main은 그대로
클로드 코드가 자동으로 브랜치를 만들기도 해요. 왜 main이 아닌 다른 브랜치인지 물어보면 이유를 설명해줘요.
main 브랜치가 특별한 이유꼭 알기

main은 "완성된, 실제 서비스에 올라가는" 브랜치예요. 여기에 직접 작업하면 실수했을 때 돌이키기 어려워요.

혼자 개발할 때: main에 직접 작업해도 사실 큰 문제는 없어요.
협업하거나 안전하게 할 때: 브랜치 → PR → merge 흐름을 써요.
merge란?핵심 개념

브랜치에서 작업한 내용을 다른 브랜치(보통 main)에 합치는 것이에요.

git checkout main # main으로 이동 git merge feature/login # feature 내용을 main에 합치기
사본에서 수정한 내용을 원본 문서에 적용하는 것이에요.
PR (Pull Request)란?핵심 개념

"내 브랜치 내용을 main에 합쳐도 될까요?"라고 요청하는 것이에요. GitHub에서 하는 작업이에요.

PR 흐름:
1. 브랜치에서 작업 완료
2. GitHub에서 PR 생성
3. diff 확인 (무엇이 바뀌었는지 검토)
4. 문제 없으면 merge 버튼 클릭
diff란?알면 좋음

변경 사항의 차이를 보여주는 것이에요. 어떤 줄이 추가됐는지(초록), 삭제됐는지(빨강).

# PR에서 이런 식으로 표시돼요 - 이 줄은 삭제됐어요 (빨간색) + 이 줄이 추가됐어요 (초록색)
push vs pull헷갈리는 쌍

push: 내 컴퓨터(로컬)의 코드를 GitHub(원격)에 올리는 것
pull: GitHub(원격)의 코드를 내 컴퓨터(로컬)로 내려받는 것

git push origin main # 로컬 → GitHub git pull origin main # GitHub → 로컬
push = 구글 드라이브에 업로드. pull = 구글 드라이브에서 다운로드.
commit이란?꼭 알기

변경 사항을 저장 + 메모하는 것이에요. 게임 세이브 포인트처럼 커밋 시점으로 언제든 돌아갈 수 있어요.

git add . # 변경된 파일 선택 git commit -m "로그인 기능 추가" # 저장 + 메모
전체 흐름 한눈에 보기정리
혼자 개발하는 가장 간단한 흐름:
코드 수정 → git add . → git commit -m "메모" → git push → 배포 자동 업데이트

브랜치를 쓰는 안전한 흐름:
새 브랜치 생성 → 코드 수정 → commit → push → GitHub에서 PR 생성 → diff 확인 → merge → main 배포
Chapter 16 · 개발 환경 실전 NEW
작업하다 갑자기 튀어나오는 개념들
클로드 코드랑 실제로 개발하다 보면 "이게 뭐지?" 싶은 단어들이 갑자기 등장해요. 브랜치·PR 정도 수준의 것들만 골라서 정리했어요.
HEAD자주 보임

지금 내가 작업하고 있는 현재 위치예요. "로컬 HEAD ↔ origin 동기화 완료"는 내 컴퓨터와 GitHub가 같은 상태라는 뜻이에요.

대부분 신경 안 써도 돼요. 클로드가 "HEAD가 어쩌구" 하면 그냥 "현재 위치" 정도로 이해하면 충분해요.
origin자주 보임

GitHub에 있는 내 레포지토리(원격 저장소)의 별명이에요. git push origin main은 "origin(GitHub)의 main 브랜치에 올려줘"라는 뜻이에요.

git push origin main # origin = GitHub 주소의 별명 git pull origin main # GitHub에서 내려받기
충돌 (Conflict)당황하지 말기

merge할 때 같은 줄을 두 곳에서 다르게 수정했을 때 발생해요. Git이 "어느 걸 써야 해요?"라고 묻는 상황이에요.

충돌이 나면 파일 안에 이런 표시가 생겨요:

<<<<<<< HEAD
내 버전의 코드
=======
상대방의 코드
>>>>>>> feature/login
충돌 났을 때 클로드에게 "충돌 났어, 파일 보여줄게"라고 하면 해결해줘요.
스테이징 영역 (git add가 왜 따로 있냐)궁금했을 것

git add는 "이 파일을 다음 커밋에 포함할게"라고 선택하는 단계예요. 수정한 파일이 10개여도 그 중 3개만 골라서 커밋할 수 있어요.

택배 포장. 물건을 박스에 넣고(add) 테이프로 봉하는(commit) 두 단계예요.
포트 (Port) — localhost:3000의 3000자주 보임

같은 컴퓨터에서 여러 서버를 동시에 실행하기 위한 번호예요.

localhost:3000 → 3000번 포트에서 실행 중인 앱
localhost:8080 → 8080번 포트에서 실행 중인 앱
"포트 3000이 이미 사용 중" 에러 = 이미 그 번호로 다른 앱이 실행되고 있다는 뜻이에요.
환경 (Environment) — dev / staging / production꼭 알기
development (dev): 개발 중인 환경. 내 컴퓨터. 막 고쳐도 됨.
staging: 배포 전 테스트 환경. 실제랑 똑같이 만들어둔 테스트 공간.
production (prod): 실제 사용자가 쓰는 환경. 신중하게 다뤄야 함.
CORS 에러자주 만남

브라우저가 다른 주소의 서버에 요청할 때 막히는 보안 정책이에요.

CORS 에러 나면 클로드에게 에러 메시지 그대로 붙여넣으면 돼요.
캐시 (Cache)알면 유용

속도를 위해 데이터를 미리 저장해두는 것이에요. "수정했는데 변경이 안 보여요" 현상의 원인이 되기도 해요.

브라우저 캐시 지우기: Ctrl+Shift+R (강력 새로고침).
토큰 (Token) vs API 키알면 좋음

API 키: 서비스에 접근하는 영구적인 비밀번호 같은 것.
토큰(Token): 로그인 후 임시로 발급되는 출입증. 보통 시간이 지나면 만료돼요.

.env 파일 vs Secrets꼭 구분하기

.env 파일: 내 컴퓨터(로컬)에서 쓰는 비밀 설정 파일. GitHub에 올리면 안 돼요.
Secrets (GitHub / Vercel): 배포 환경에서 쓰는 비밀 설정.

.gitignore 파일에 .env가 들어 있어야 GitHub에 안 올라가요.
CRUD꼭 알기

데이터를 다루는 4가지 기본 동작의 앞글자를 딴 말이에요.

Create — 만들기 (게시글 쓰기, 회원가입, 할 일 추가)
Read — 읽기 (목록 불러오기, 상세 페이지 보기)
Update — 수정하기 (게시글 편집, 프로필 변경)
Delete — 삭제하기 (게시글 지우기, 계정 탈퇴)
노션 데이터베이스로 비유하면: 행 추가(C) / 내용 보기(R) / 셀 수정(U) / 행 삭제(D)예요.

브랜치 실전 생존 가이드
브랜치 때문에 작업 내용이 유실되거나, 어느 브랜치에 있는지 몰라 혼란스러웠던 경험 — 이 챕터 하나로 정리해요.
지금 내가 어느 브랜치에 있는지 항상 확인하기가장 중요

작업 시작 전에 현재 브랜치를 확인하는 습관이 가장 중요해요. 모르고 엉뚱한 브랜치에서 작업하면 나중에 꼬여요.

git branch # 브랜치 목록 + 현재 위치(* 표시) git status # 현재 브랜치 + 변경된 파일 목록
클로드 코드가 브랜치를 자동으로 만드는 이유꼭 알기

클로드 코드는 작업할 때 자동으로 새 브랜치를 만들기도 해요. main을 보호하고 작업 단위를 분리하기 위해서예요.

클로드 코드가 새 브랜치를 만들었다면 작업 후 반드시 "이 브랜치 main에 merge해줘"라고 말해야 해요.
작업 내용이 사라진 것처럼 보일 때유실 사고 원인 1

브랜치를 이동하면 파일 내용이 그 브랜치 상태로 바뀌어요. 사라진 게 아니라 다른 브랜치에 있는 거예요.

# 내가 작업하던 브랜치로 다시 이동 git checkout feature/내가-작업하던-브랜치 # 브랜치 목록 모를 때 git branch
커밋 안 하고 브랜치 이동하면 생기는 일유실 사고 원인 2
안전한 브랜치 이동 순서:
1. 작업 완료
2. git add . → git commit -m "작업 내용" (저장 먼저)
3. 그 다음 git checkout 다른브랜치
커밋하지 않고 브랜치를 이동하는 건 위험해요. 이동하기 전에 항상 커밋하거나, stash를 쓰세요.
stash — 임시 저장유용한 도구

커밋하기엔 애매한데 일단 작업 내용을 잠깐 치워둘 때 쓰는 임시 보관함이에요.

git stash # 현재 변경사항 임시 저장 (작업 공간 깨끗해짐) git stash pop # 임시 저장한 내용 다시 꺼내기
서랍에 잠깐 넣어두기. 나중에 꺼내면 그대로 있어요.
reset — 가장 위험한 명령어위험
git reset --soft: 커밋만 취소, 파일 변경사항은 남아있음 (비교적 안전)

git reset --mixed: 커밋 취소 + add도 취소, 파일은 남아있음

git reset --hard: 커밋 취소 + 파일 변경사항까지 전부 삭제 (위험!)
클로드 코드가 reset을 하려고 할 때 어떤 종류인지, 어떤 커밋으로 돌아가는지 꼭 물어봐요.
작업 흐름 — 클로드 코드와 안전하게 쓰는 패턴추천 루틴
매 작업 시작 전
→ "지금 브랜치 상태 확인해줘" (현재 위치 파악)

기능 하나 만들 때
→ "새 브랜치 만들어서 [기능] 작업 시작해줘"

작업 중간중간
→ "지금까지 작업 커밋해줘" (중간 저장)

기능 완성됐을 때
→ "main에 merge하고 브랜치 정리해줘"

세션 끝내기 전
→ "커밋 안 된 내용 있으면 커밋하고 push까지 해줘"
세션을 그냥 닫기 전에 반드시 커밋·push 여부를 확인해요. 이게 유실 사고의 거의 모든 원인이에요.
이 상황 전체를 베이 언어로 정리스크린샷 해석
"브랜치 분기 문제가 있네요. merge로 해결하고, tool로 진행할게요."
→ 브랜치가 꼬여있어서 먼저 합치는 작업(merge)을 하고, 그 다음에 파일을 직접 수정할게요.

"워킹 트리가 깨끗하네요."
→ 커밋 안 된 변경사항은 없어요. 일단 안전한 상태예요.

"로컬 main이 origin/main과 완전히 다른 히스토리를 가지고 있어요"
→ 내 컴퓨터의 코드 기록이랑 GitHub의 코드 기록이 서로 달라요.

"로컬을 origin/main으로 리셋할게요."
→ GitHub 기준으로 내 컴퓨터를 강제로 맞출게요. ⚠️ 로컬에만 있던 작업 내용은 사라져요.
다음번에 클로드 코드가 "리셋할게요"라고 하면 — 바로 진행하지 말고 "리셋하면 사라지는 내 작업이 있어?" 먼저 물어봐요.
